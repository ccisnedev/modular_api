# Migrating from 0.4.x to 0.5.0

What changes for consumers moving to the 0.5.0 line, and what to check while
upgrading. Versions are synchronized across the TypeScript, Dart, and Python SDKs
(ADR-0002); TypeScript ships the 0.5.0 features first, Dart and Python reach
parity within the synchronized release.

## 1. Operational endpoints live under basePath

If you are coming from an early 0.4.x (<= 0.4.4), health, metrics, docs, and the
OpenAPI spec are no longer at the server root â€” they resolve under the configured
`basePath` (this landed in 0.4.7):

| Before (<= 0.4.4) | Now |
|---|---|
| `/health` | `{basePath}/health` |
| `/metrics` | `{basePath}/metrics` |
| `/docs` | `{basePath}/docs` |
| `/openapi.json` / `.yaml` | `{basePath}/openapi.json` / `.yaml` |

Action: update container healthchecks, Prometheus scrape configs, and any
reverse-proxy rules. See [operational-plugins.md](../concepts/operational-plugins.md).

## 2. PluginRoute.openapi â€” plugin routes in /docs and /openapi.json (ADR-0003)

`PluginRoute` gains an optional `openapi` field holding a standard OpenAPI
Operation object. Routes with visibility `custom` or `transport` that declare it
now appear in the generated spec and in `/docs` â€” including binary endpoints
(`image/jpeg` with `schema: { type: string, format: binary }`).

- Existing plugins remain valid: the field is optional; undocumented routes simply
  do not appear in the spec.
- `PluginHost.routes()` is new: a read view of registered plugin routes (method,
  absolute mounted path, visibility, openapi operation).

Action (optional but recommended): add `openapi` operations to consumer-facing
plugin routes so the generated documentation is complete. See
[plugin-host.md](../concepts/plugin-host.md).

## 3. Plugin routes get real metric labels

Plugin route paths are now part of the metrics route registry. Requests to plugin
routes are labeled with their real route in `http_requests_total` and
`http_request_duration_seconds` instead of `UNMATCHED`.

Action: if you built dashboards or alerts around `route="UNMATCHED"` as a proxy
for plugin traffic, update them to the real route labels.

## 4. basePath convention in the package manifest

0.5.0 standardizes where deployment tooling reads the service basePath:

- TypeScript: `package.json` -> `"modularApi": { "basePath": "/api/v1" }`
- Dart: `pubspec.yaml` -> `modular_api:` section
- Python: `pyproject.toml` -> `[tool.modular_api]`

`Publish-NodeApi` (macss-devops >= 3.1.0) resolves the healthcheck URL with
precedence `package.json` > `publish.yaml api.basePath` > server root.

Action: declare the convention in your manifest. See
[deployment.md](../guides/deployment.md).

## 5. Stage 3 decorators â€” remove experimentalDecorators (TypeScript)

The `@Field` decorators are TC39 Stage 3 decorators. Consumer projects must NOT
enable `experimentalDecorators` in any tsconfig:

- With the flag on, field metadata is never registered and DTOs fail at runtime
  with `<Name>.toJson() not implemented`.
- Under vitest, the equivalent failure appears as a `SyntaxError` at import time
  or a `TypeError` mentioning `Symbol(FieldMeta)` when using rolldown-vite/OXC.

The verified test setup is vitest 4.0.x with standard vite 7 and zero transform
configuration (closes issue #19).

Action: delete `experimentalDecorators` from every tsconfig; see
[pitfalls.md](../pitfalls.md) and the [testing guide](../guides/testing.md).

## Checklist

- [ ] Healthchecks and scrapes point to `{basePath}/health` and `{basePath}/metrics`
- [ ] Consumer-facing plugin routes declare `openapi` operations
- [ ] Dashboards updated for real plugin route labels
- [ ] basePath convention declared in the package manifest
- [ ] No `experimentalDecorators` in any tsconfig
- [ ] Test runner on vitest 4.0.x / vite 7 (not rolldown-vite)
